Title: A Query-Aware Sentinel-2 Time-Series Model for Crop Type and Rice Phenology Prediction

Draft status: pre-submission article base
Primary submission described here: C03 full-data model with the P6 fast inference pipeline
Timing and delay metrics: platform metrics are included for C03, C20, P3, and final C03 + P6


1. Executive Summary

This project solves Track 1 of the 2026 Artificial Intelligence and Space Computing Challenge, "Space Intelligence Empowering Zero Hunger." The task is to predict, for hidden test points, both the crop category and the phenological stage associated with a requested date. The input is a platform-mounted inference dataset containing a point CSV and multi-temporal Sentinel-2 TIFF imagery. The output must be a single JSON file at /output/result.json.

The final submission uses a compact query-aware CNN + Transformer model, identified internally as C03 full-data epoch 75. The model takes a multi-date Sentinel-2 patch time series around each point, adds acquisition day-of-year information, adds the requested query day-of-year, and predicts:

- crop type: corn, rice, or soybean
- phenophase stage: Greenup, MidGreenup, Peak, Maturity, MidSenescence, Senescence, or Dormancy

The final runtime package uses the P6 preprocessing and inference pipeline. P6 is not a different model; it is the faster execution path around the C03 model. It keeps the same model weights and prediction behavior, but improves inference-side data preparation through fast TIFF inventory, GDAL configuration, in-memory patch tensors, no NPZ write during inference, multi-worker raster extraction, and bounded batched raster window reads.

The final package intentionally does not use XGBoost, TSViT, model ensembles, hard stage bijection, or transition/Viterbi postprocessing. Those were investigated as comparison or high-upside alternatives, but the final operational choice is the simpler C03 neural model because it is small, validated, easier to explain, and safer for a constrained inference-only space runtime.


2. Competition Objective and Output Contract

The official task asks participants to use multi-temporal Sentinel-2A/Sentinel-2B multispectral surface reflectance imagery during the rice growing season to predict hidden point labels. The final-round test data is not public and is mounted by the platform under /input.

The expected Track 1 input structure is:

/input/test_point.csv or /input/points_test.csv
/input/region_test/*.tiff

The required output is:

/output/result.json

The JSON key must be formed from longitude, latitude, and query date:

{Longitude}_{Latitude}_{Date}

The JSON value must be a two-element list:

[CropType, PhenophaseStage]

An example output row is:

"124.703696_48.543523_2018/9/1": ["corn", "Senescence"]

The scoring rule combines two macro-F1 terms:

competition_score = 0.4 * crop_macro_f1 + 0.6 * rice_phenology_macro_f1

The crop score is a macro-average F1 over the three target crop classes. The rice phenology score is stricter: it evaluates rice-stage predictions only through a "double hit" condition. A prediction is counted as correct for a rice phenology stage only when both the crop label is rice and the phenological stage label is correct. This makes the model's rice crop recognition and rice stage prediction tightly coupled. A crop mistake can damage the rice phenology score even if the predicted stage label looks plausible.

This scoring rule strongly influenced the model design. The pipeline must be reliable at recognizing crop type, but it must also answer the date-specific phenology question for rice. A model that knows only the spatial crop pattern but ignores the requested date is insufficient.


3. Data Representation

3.1 Sentinel-2 Bands

The model uses the standard 12-band Sentinel-2 order used throughout the project:

B01, B02, B03, B04, B05, B06, B07, B08, B8A, B09, B11, B12

Each band is stored as a separate TIFF file for a region and acquisition date. The preprocessing code parses filenames to identify region, acquisition date, processing level, and band. Files that do not match the expected Sentinel-2 naming pattern are rejected. Unsupported bands are ignored.

When duplicate candidates exist for the same region, date, and band, the file selection is deterministic:

- prefer valid supported Sentinel-2 bands
- prefer L2A over L1C when multiple processing levels exist
- use path order as a deterministic tie breaker

This deterministic file selection is important for reproducibility: the same mounted input directory should always generate the same patch tensor.

3.2 Point Rows and Query Rows

The training labels contain:

- point_id
- Longitude
- Latitude
- phenophase_date
- crop_type
- phenophase_name

A point represents a fixed spatial location, but it can have multiple phenophase dates. The model treats each point-date pair as a separate query. The same image time series can therefore be used multiple times with different query dates. This is the central modeling idea: crop type is mostly spatial and seasonal, while phenophase stage is date-dependent.

The inference code accepts column-name variants such as Longitude, Longtitude, longitude, lon, Latitude, lat, phenophase_date, Date, date, query_date, datetime, time, point_id, pointid, and id. This protects the submission from minor platform-side naming variation.

3.3 Region Mapping

The preprocessing stage builds a region catalog from selected TIFFs. For each point, the code checks which raster region bounding boxes contain the point. If exactly one region contains the point, that region is used. If multiple regions contain the point, the code chooses the best region by:

1. highest number of time steps
2. highest band count
3. largest border margin
4. region ID as deterministic tie breaker

In training mode, points outside all regions are dropped. In inference mode, nearest-region fallback is allowed so the platform can still produce predictions for all test points.

3.4 Patch Extraction

For every selected point, acquisition date, and band, the preprocessing code extracts a 15 x 15 pixel patch centered on the point. The patch size is odd, so there is one center pixel and seven pixels of context in each direction.

The longitude/latitude to pixel conversion is:

pixel_x = round((lon - xmin) / pixel_size_x)
pixel_y = round((ymax - lat) / pixel_size_y)

If the patch would extend outside the TIFF boundary, the point is not discarded. The valid part of the raster window is read, and missing rows or columns are filled by edge replication. Edge replication repeats the nearest real edge values instead of inserting zeros. This keeps border points usable while avoiding artificial zero borders.

The preprocessing report records whether a patch needed edge replication and whether the center had to be clamped into the raster.

3.5 Invalid Pixel Handling

A pixel is considered valid only when:

finite and 0.0 < value <= 2.0

Invalid pixels are handled in two ways:

1. the patch value is filled with 0.0
2. a valid_pixel_mask records which pixels were valid

This distinction matters because the model uses valid-mask channels. A zero value alone would be ambiguous, but a separate mask channel tells the model whether a pixel is an actual valid measurement or a filled invalid value.

3.6 Tensor Shape

The cleaned training tensor has the schema:

patches: [N, T, 12, 15, 15]
valid_pixel_mask: [N, T, 12, 15, 15]
time_mask: [N, T]
time_doy: [N, T]
band_mask: [N, T, 12]

For the cleaned training artifact, N is 778 unique points and the maximum T is 29 acquisition dates. Different regions have different date counts, so shorter sequences are padded. The time_mask identifies real acquisition dates and prevents padded dates from affecting the Transformer.

For the final C03 model, the model input channel count is 24:

- 12 normalized Sentinel-2 bands
- 12 binary valid-mask channels

3.7 Normalization

The active normalization is z-score normalization computed only from valid pixels in the training data:

normalized = (raw_value - training_band_mean) / training_band_std

After normalization, invalid pixels are reset to 0.0. The normalization statistics are stored in:

artifacts/normalization/train_patch_band_stats.json

The submission package includes this JSON and validates that it contains 12 bands and per-band mean/std values before inference.


4. Training Query Construction

The NPZ stores one image time-series tensor per unique point. The PyTorch dataset expands that point tensor into query rows. For each point:

1. read the crop label
2. read the seven phenophase day-of-year labels
3. create one query for each known phenophase date
4. attach the full point time series to every query
5. attach the requested query_doy
6. set the crop target to the point crop
7. set the stage target to the queried phenophase stage

The final full-data C03 model was trained on all labeled query rows after the validation evidence had selected epoch 75. The full-data training run used 5,446 query rows.

5.3 Per-Date CNN Encoder

Each acquisition date is encoded separately by the same small CNN. The CNN receives one date's patch tensor with 24 channels and spatial size 15 x 15. Its structure is:

1. Conv2D from input channels to 32 channels
2. BatchNorm
3. GELU
4. Conv2D from 32 to 64 channels
5. BatchNorm
6. GELU
7. MaxPool2D
8. Conv2D from 64 to 96 channels
9. BatchNorm
10. GELU
11. Adaptive average pooling to one spatial cell
12. Flatten
13. Dropout
14. Linear projection to 256 dimensions
15. LayerNorm

The output is one 256-dimensional embedding per acquisition date.

5.4 Acquisition Day Encoding

The model uses acquisition day-of-year information through a sinusoidal day encoding. The day-of-year is converted into sine and cosine seasonal features and projected into the Transformer dimension. This encoding is added to the per-date CNN embedding, giving the Transformer information about when each satellite observation occurred.

This is important because two visually similar patches can mean different things depending on whether they occur early, mid, or late in the growing season.

5.5 Temporal Transformer

The date embeddings are processed by a Transformer encoder:

- Transformer dimension: 256
- Transformer layers: 4
- Attention heads: 8
- Feed-forward dimension: 1024
- Activation: GELU
- Dropout: 0.18

The Transformer receives a key padding mask derived from time_mask, so padded dates do not influence attention.

5.6 Masked Temporal Pooling

After the Transformer, the model uses masked temporal average pooling. Only real acquisition dates are pooled. Padded dates are ignored. The result is one seasonal representation vector for the point.

5.7 Query Day Encoding

The query day-of-year is encoded separately using the same seasonal logic. This creates a 256-dimensional query vector. The model concatenates:

- pooled seasonal image representation
- query date representation

This is the core reason the model can answer different phenophase questions for the same spatial point. If the query date changes, the input image time series can stay the same while the predicted stage changes.

5.8 Output Heads

C03 has two output heads:

- crop head: LayerNorm, Dropout, Linear to 3 crop classes
- stage head: LayerNorm, Dropout, Linear to 7 phenophase classes

The model returns crop logits and stage logits for every query row.


6. Training Objective and Metrics

The training loss is:

total_loss = crop_loss + 0.6 * stage_loss

Crop loss is cross entropy over the three crop classes and is applied to every query row. Stage loss is cross entropy over the seven phenophase stages. In the active configuration, rice_stage_loss_only is true, so the stage loss is counted only for rice rows. This matches the competition objective, where the phenology score is rice-specific.

The model still outputs stage logits for corn and soybean rows because the required result JSON always contains both crop and stage labels. However, the training loss emphasizes rice stage correctness because that is the scored phenology target.

The project records:

- crop accuracy
- crop macro-F1
- crop consistency metrics
- stage accuracy over all crops
- rice-stage accuracy
- rice-stage macro-F1
- joint accuracy
- competition score

The competition score used internally is:

0.4 * crop_macro_f1 + 0.6 * rice_stage_macro_f1


7. Final Inference Pipeline: C03 With P6 Runtime

7.1 Runtime Entry Point

The submission image starts from:

./run.sh

The script:

1. prints runtime diagnostics
2. detects Python and PyTorch
3. resolves INPUT_ROOT, OUTPUT_DIR, and WORK_DIR
4. chooses CANDIDATE=p6 by default
5. selects configs/submission_c03_p6.json
6. verifies the platform input files
7. writes a runtime config into /tmp
8. validates the package
9. runs inference.py

The final package accepts either:

/input/test_point.csv

or:

/input/points_test.csv

7.2 P6 Configuration

The active P6 config uses:

- checkpoint: checkpoints/c03_full_data_model.pt
- crop_checkpoint: checkpoints/c03_full_data_model.pt
- stage_checkpoint: checkpoints/c03_full_data_model.pt
- normalization_json: artifacts/normalization/train_patch_band_stats.json
- batch_size: 64
- device: auto

- batch_raster_reads: true

P6 is designed to make the same C03 model run with faster preprocessing. It does not change the trained weights. It does not add XGBoost. It does not add a second model. It does not apply hard stage ordering.

7.3 In-Memory Patch Extraction

Earlier inference paths could write an intermediate test_patches.npz. The P6 path avoids that write by requesting in-memory arrays from build_patch_dataset. This reduces filesystem work and keeps the pipeline simpler during submission inference.

7.4 Fast TIFF Inventory

The fast inventory path keeps the same file-selection rules as the full audit path, but avoids building the complete training-style report dataframe. This is appropriate for inference because the platform only needs predictions, not diagnostic CSV reports.

7.5 GDAL and Raster I/O Configuration

The P6 config sets GDAL options for many small local window reads:

- disable expensive sibling-directory scans on open
- set read-directory limit to zero
- increase GDAL cache
- enable VSI cache

Raster extraction can group nearby patch centers into bounded union windows. The goal is to reduce repeated tiny reads while avoiding excessive overread. The configured limits are:

- max_batch_union_pixels = 262144
- max_batch_union_overread_ratio = 6.0

If a clustered union would read too much extra data, the code falls back to exact per-patch reads.

7.6 Device Selection

The config uses device=auto. The inference code selects CUDA if torch.cuda.is_available() is true; otherwise it uses CPU. This gives the same container a GPU path on the competition platform and a CPU fallback for local or constrained environments.

7.7 Crop Consistency

The same spatial point can appear in several query rows because it is asked about several dates. Crop type should not change across dates for the same point. The final inference path therefore applies crop consistency:

1. group query rows by point_id
2. sum crop logits across all query rows for that point
3. choose the crop with the largest summed logit
4. assign that crop to all query rows of the point

This preserves the model's evidence while enforcing a semantic constraint.

7.8 Output-Key Consistency

The final result JSON is keyed by longitude, latitude, and date. If duplicate input rows collapse to the same output key, JSON cannot represent conflicting duplicate keys. The inference code detects duplicate output keys and forces identical crop and stage predictions for all rows sharing the key. If there is disagreement, it resolves by majority vote and uses summed logits as a tie breaker.

This prevents arbitrary "first row wins" behavior and makes duplicate handling explicit.

7.9 Stage Postprocessing

The final C03 + P6 submission uses:

stage_postprocess = none
use_point_stage_bijection = false

Several stage-ordering ideas were studied, including monotonic decoding and hard point-stage bijection. They were not used in the final model because validation evidence and hidden-test behavior did not justify a hard biological ordering override. The model's neural stage logits are used directly.

7.10 No XGBoost in the Final Package

The final C03 + P6 package intentionally removes XGBoost:

- no xgboost dependency in requirements.txt
- no xgb_stage_model key in submission_c03_p6.json
- no XGBoost pickle artifact in the package
- no XGBoost stage override code in the final track1_kybelix submission script

This matters because the goal of the current submission is to test the C03 model with the faster P6 preprocessing path, not to rerun the previous XGBoost-hybrid idea.

9.4 C03 Platform Baseline Submission

The C03 full-data baseline was already submitted successfully before the final P6 cleanup. The recorded hidden/test score for that earlier C03 submission was 0.994132. Component labels from the platform table were ambiguous, so crop macro-F1 and rice-stage macro-F1 should be confirmed from a platform export before being cited separately.

9.5 C03 P3 Optimized Pipeline

P3 was an earlier runtime optimization around the C03 model. It introduced several inference-side improvements:

- fast_inventory
- use_gdal_env
- use_in_memory_patches
- write_patch_npz=false
- compressed_patch_npz=false
- crop consistency

P3 demonstrated that the model forward pass was not the main bottleneck; raster extraction dominated runtime. This motivated P6, which keeps the C03 model but adds more aggressive raster-window batching and multi-worker I/O. The article should discuss this as an engineering lesson, not as a new model.

9.6 C04 Auxiliary Feature Model

C04 added a 135-dimensional auxiliary feature vector. The auxiliary vector included per-band summaries, vegetation-index summaries, query date features, nearest acquisition distance, nearest-date band medians, and nearest-date vegetation index values.

C04 evidence:

- best validation competition score: 0.995363
- best validation crop macro-F1: 1.000000
- best validation rice-stage macro-F1: 0.992272
- best validation loss: 0.010718

C04 showed that engineered features can improve confidence/loss in some settings, but its rice-stage F1 was not better than C03. It was therefore not selected as the final model.

9.7 C07 No Query Date Ablation

C07 removed query day-of-year while keeping acquisition day enabled.

C07 evidence:

- best validation competition score: 0.436115
- best validation crop macro-F1: 0.989268
- best validation rice-stage macro-F1: 0.067347
- best validation loss: 1.111821

This is one of the most important ablations. Crop prediction stayed relatively strong, but phenology collapsed. This directly proves that the requested query date is essential for stage prediction.

9.8 C08 No Acquisition Date Ablation

C08 removed acquisition day-of-year while keeping query day enabled.

C08 evidence:

This shows that acquisition date is less critical than query date, but still useful for confidence and calibration. The final model keeps acquisition day encoding enabled.

- best epoch: 77
- best validation competition score: 1.000000
- best validation crop macro-F1: 1.000000

9.11 C29 TSViT Comparison Model

C29 TSViT is the high-capability comparison architecture.

C29 full-data evidence:

- model type: query_tsvit
- checkpoint size: 88.51 MB
- parameter count: 23,170,666
- fixed epoch: 42
- train queries: 5,446
- final train competition score: 1.000000
- final train loss: 0.166929
- input channels: 24
- patch size: 15
- token patch size: 3

- spatial layers: 6

TSViT factorizes temporal and spatial attention. It creates spatial tokens from the patch, applies temporal encoding, uses temporal attention per spatial token, and then uses spatial class tokens for crop and stage heads. It is architecturally richer than C03, but much heavier. The article should present it as a future/higher-compute path rather than the final low-risk package.

9.12 C12 Hybrid C04 Crop + C03 Stage

C12 combined crop prediction from a C04 full-data checkpoint and stage prediction from a C03 full-data checkpoint. The motivation was to combine a stronger crop/general public signal from C04 with a stronger rice phenology signal from C03. This was a useful hybrid design, but it increases package complexity and was not selected for the final C03 + P6 submission.

9.13 Hybrid C20 Fourier + Full-Data XGBoost

The hybrid C20 + XGBoost package used a neural C20 Fourier checkpoint for crop prediction and an XGBoost model for rice-stage override. Validation evidence before full-data retraining was high, including a hybrid validation score of 0.997683 and strong XGBoost rice-stage macro-F1 on the validation split.

This path was high-upside but riskier. The final C03 + P6 submission deliberately removes it. The article can mention it as a studied alternative and as evidence that tabular feature engineering was investigated, but it should not describe XGBoost as part of the final submitted model unless the submission changes again.


10. On-Orbit Feasibility

10.1 Uplink Package

The final package contains:

- source code
- run.sh
- Dockerfile
- requirements.txt
- config JSON
- train-derived normalization JSON
- C03 checkpoint

The final package excludes:

- raw training TIFFs
- training artifacts
- unused checkpoints
- XGBoost artifacts
- TSViT checkpoint

The C03 checkpoint is 12.55 MB. The code and configuration are small compared with the checkpoint. The PyTorch/CUDA base image is provided by the platform, so the submitted repository does not need to include PyTorch itself.

10.2 Runtime Environment

The official PyTorch base environment provides:

- Ubuntu 22.04 base
- PyTorch 2.5.1 with CUDA 12.1
- cuDNN 9
- Python 3.11-compatible runtime
- rasterio, numpy, pandas, scipy, torchvision, torchaudio, and related core libraries

The final requirements.txt is minimal:

- numpy
- pandas
- rasterio

PyTorch is not installed manually by the submission because the base image already provides it.

10.3 Compute Strategy

The pipeline naturally separates CPU and GPU work:

- CPU: file discovery, TIFF metadata, point mapping, raster window reads, patch cleaning, normalization
- GPU when available: C03 neural inference
- CPU fallback: supported through device=auto

This supports heterogeneous scheduling in an on-orbit or edge setting. The model is small enough that CPU fallback remains possible, while GPU acceleration can be used when available.

10.4 Memory and Storage

The model itself is compact. The largest temporary memory object is the extracted patch tensor. The tensor size scales with:

batch_size x time_steps x channels x patch_height x patch_width

For C03, channels are 24 after adding valid-mask channels. The patch size is fixed at 15 x 15. The sequence length depends on the region's available acquisition dates. The final P6 path avoids writing a test NPZ unless explicitly configured, reducing temporary disk use.

10.5 Fault Tolerance and Validation

The pipeline includes several fault-tolerance features:

- accepts both test_point.csv and points_test.csv
- validates config and checkpoint before inference
- supports CPU fallback
- handles duplicate output keys deterministically
- allows nearest-region fallback in inference mode
- records file read failures and patch extraction errors
- uses deterministic file selection
- does not download external data at inference time

11. Innovation of the Implementation Path

The main innovation is not a single exotic model component. It is the complete query-aware, deployment-focused design:

1. Multi-temporal point-centered Sentinel-2 patch tensor construction
2. Explicit valid-mask channels for remote-sensing missing/invalid pixels
3. Query-day conditioning for date-specific phenology
4. Separate crop and stage heads sharing a seasonal representation
5. Rice-specific stage loss aligned with the scoring objective
6. Crop consistency across query dates for the same point
7. Deterministic output-key conflict resolution
8. Fast inference-oriented raster extraction
9. Small final checkpoint suitable for constrained deployment
10. Larger TSViT and Fourier variants kept as research comparisons

Compared with a ground-only workflow, this design emphasizes moving compact models and compact predictions instead of raw imagery. The satellite-side or edge-side system can read local imagery, produce crop and phenology status, and downlink only the JSON result or aggregated products. This reduces bandwidth demand and supports faster decision-making.

The C03 final model is intentionally conservative. The innovation story is strengthened by the experimental record: TSViT, Fourier time encoding, auxiliary features, and XGBoost were all investigated, but the final system chooses the smallest reliable model for operational submission.


12. Value and Application Scenarios

The practical value is agricultural monitoring from space imagery. The system can support:

- rice growth-stage tracking
- corn, rice, and soybean classification
- crop-season progress monitoring
- food-security analysis
- regional agricultural reporting
- early warning for abnormal seasonal development
- satellite-terrestrial coordination where orbit-side inference filters large imagery streams before ground analysis

For decision makers, the output is not a raw image. It is a structured crop and phenology product tied to points and dates. This makes it easier to integrate into dashboards, reports, crop calendars, and downstream agricultural analytics.

15. Short Article-Style Explanation

Our solution is built around a simple observation: crop type is mostly a property of the location and its seasonal image pattern, but phenological stage is a property of the location at a requested date. A model cannot reliably predict rice phenology unless it knows the query date. Therefore, we designed a query-aware Sentinel-2 time-series model.

For each point, the preprocessing system finds the relevant Sentinel-2 TIFF files, chooses one file per region, date, and band, maps the longitude and latitude into raster coordinates, and extracts a 15 x 15 patch for each of 12 Sentinel-2 bands across all available acquisition dates. Invalid pixels are filled but also recorded in a valid mask. The final model receives both the normalized spectral values and the mask channels, so it can distinguish missing or invalid observations from real reflectance.

Each acquisition date is encoded by a small CNN. The model adds acquisition day-of-year information, processes the full sequence with a Transformer, then pools only valid acquisition dates. A separate query day-of-year encoding tells the model which date the phenology question is asking about. The model combines the seasonal image representation and the query-date representation to predict both crop type and phenophase stage.

Training uses crop cross entropy for all query rows and rice-stage cross entropy for rice rows, matching the competition's scoring focus. The final model was selected because it reproduced the earlier high-confidence C00 baseline, achieved perfect validation crop and rice-stage macro-F1 in the validation evidence, and remained compact enough for constrained deployment. The final full-data checkpoint uses all labeled query rows and is only 12.55 MB.

The final submission uses the C03 model inside the P6 inference pipeline. P6 keeps the same model weights but improves inference-side data preparation with fast TIFF inventory, GDAL configuration, in-memory patch arrays, no intermediate NPZ write, multi-worker I/O, and bounded raster-window batching. The package writes exactly /output/result.json and includes validation checks before running inference. The final package does not use XGBoost, does not use TSViT, and does not use a model ensemble.

The broader experiment history supports the final design. Removing query date caused rice-stage prediction to collapse, proving that query conditioning is essential. Removing acquisition date kept much of the score but worsened confidence, so acquisition timing remains useful. Removing valid-mask channels hurt performance, so mask channels remain part of the final input. Fourier temporal encoding and TSViT were valuable innovation paths, but the final operational submission prioritizes the smaller and safer C03 model.

The result is an end-to-end satellite inference system: platform input is mounted, TIFFs are transformed into point-centered time-series tensors, a compact query-aware neural model predicts crop and phenology labels, consistency logic resolves semantic and JSON-key constraints, and the final product is downlinked as a compact result JSON.
